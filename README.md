# MedEdge — Private, Offline Medical Q&A on the Edge

**A fully on-device medical question-answering assistant powered by [QVAC](https://qvac.tether.io) MedGemma, with peer-to-peer delegated inference. Zero cloud. Your health questions and documents never leave your hardware.**

Built for the **QVAC Hackathon I — Unleash Edge AI**. Tracks: **Psy Models** (QVAC MedGemma medical model) and **General Purpose** (runs on a ≤32 GB consumer machine).

License: **Apache-2.0**. All AI inference runs through the **QVAC SDK** (`@qvac/sdk`) — no cloud AI API is ever called.

---

## What it is

MedEdge answers natural-language health questions using **retrieval-augmented generation (RAG)** over a local corpus of medical reference notes, with answers generated by **MedGemma-4B** (a medical instruction-tuned model) running on your own GPU. Two modes:

1. **Local mode** — retrieval *and* generation run on this device. Works in airplane mode after the one-time model download.
2. **P2P mode** — a weak "consumer" device runs the lightweight retrieval **on-device** and **delegates the heavy MedGemma generation to a stronger "provider" peer** over the Hyperswarm DHT (Holepunch). This is real edge **load distribution**: the small embedding model stays local; only the expensive generation is offloaded.

Every run writes a structured, auditable performance log (model loads, prompt, tokens, TTFT, tokens/sec, GPU/CPU) to `logs/`.

## Why it matters

Health data is among the most sensitive data there is. Cloud chatbots send your symptoms and documents to someone else's servers. MedEdge keeps everything **on hardware you own** — on-device, or delegated only to a peer device *you* control. It demonstrates that private, production-useful edge AI is possible today: a 4B medical model answering grounded questions at **70–120 tokens/sec on a laptop GPU**, with no cloud dependency.

## Architecture

```
                         ┌─────────────────────────────────────────┐
   question ─────────────▶            MedEdge assistant             │
                         │                                          │
                         │  1. RAG retrieval  (ALWAYS on-device)    │
                         │     GTE-large embeddings + QVAC built-in │
                         │     vector store → ragSearch(topK)       │
                         │                                          │
                         │  2. Grounded generation                  │
                         │     MedGemma-4B via QVAC completion()    │
                         │        ├─ local mode: this GPU           │
                         │        └─ P2P mode:  delegate ──────────────▶  Provider peer
                         │                     (Hyperswarm DHT)     │       (runs MedGemma,
                         │  3. Audit log (TTFT, tok/s, tokens, GPU) │        returns tokens)
                         └─────────────────────────────────────────┘
```

## QVAC SDK usage (full-stack)

| Capability | QVAC SDK API used |
|---|---|
| On-device LLM generation | `loadModel` / `completion` (streaming) / `unloadModel` |
| Embeddings | `loadModel(GTE_LARGE_FP16)` + RAG embed pipeline |
| RAG with built-in vector store | `ragIngest` / `ragSearch` (no external DB) |
| P2P delegated inference | `startQVACProvider` + `loadModel({ delegate })` over Hyperswarm |
| Performance telemetry | `completion().stats` → `timeToFirstToken`, `tokensPerSecond`, `promptTokens`, `generatedTokens`, `backendDevice` |
| Model config | `modelConfig: { ctx_size, temp, verbosity }` |

No cloud AI API is used. See [`evidence/remote-apis.json`](evidence/remote-apis.json).

## Measured performance

Measured on the included hardware (NVIDIA RTX 5080 Laptop GPU, Vulkan 1.4; full specs in [`evidence/hardware.md`](evidence/hardware.md)), MedGemma-4B-it Q4_1, warm:

| Mode | TTFT (warm) | Throughput | Backend |
|---|---|---|---|
| Local | ~137–147 ms | 71–114 tok/s | GPU (Vulkan) |
| P2P delegated (served by provider) | ~147–189 ms | 118–121 tok/s | GPU (Vulkan) |

The **first** inference in a process incurs a one-time ~5 s Vulkan shader-compilation cost; the app runs a `warmup()` pass so measured questions reflect warm steady state. Both cold and warm events are visible in the audit logs. The first P2P connection on a cold DHT can take 15–45 s (Hyperswarm bootstrap + holepunch); subsequent calls are sub-second.

## Requirements

- **Node.js ≥ 22.17** (developed on v24.17.0)
- A **Vulkan-capable GPU** on Windows/Linux, or Apple Metal on macOS (see the [QVAC compatibility matrix](https://docs.qvac.tether.io/installation/)). CPU fallback works but is slower.
- ~4 GB free disk for cached models.

## Install & run

```bash
npm install
```

**Local mode** (retrieval + generation both on this device):

```bash
npm run ask                                  # runs 3 demo questions
npm run ask -- "What lifestyle changes lower blood pressure?"
```

**P2P mode** (two terminals — the provider can be a second machine you own):

```bash
# Terminal 1 — the strong device that runs MedGemma for peers:
npm run provider                             # prints a provider public key

# Terminal 2 — the weak device; retrieval is local, generation is delegated:
npm run consumer -- <provider-public-key>
```

First run downloads **GTE-large** (~670 MB) and **MedGemma-4B** (~2.56 GB) once via the QVAC P2P model registry, then runs fully offline.

> **Proving the delegation is real:** the consumer defaults to `fallbackToLocal=false`, so it can only answer if the provider genuinely served the inference. The provider's [`evidence/provider-session.log`](evidence/provider-session.log) shows the matching `kind=completion` requests it handled. Set `QVAC_FALLBACK=true` to allow a silent local fallback for demo robustness.
